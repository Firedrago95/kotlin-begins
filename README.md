# 코틀린 학습

## 📝 [1일차] 객체지향과 엔티티 설계
*   **Top-level Function:** 유틸/진입점 용도. 자바의 무의미한 `Utils` 클래스 대체.
*   **Companion Object:** 도메인 결합 정적 메서드(`create()`) 캡슐화.
*   **`val` / `var`:** 기본은 무조건 `val`(참조 불변성). 참조가 아예 바뀔 때만 `var`.
*   **주 생성자(Primary Constructor):** 필드 선언, 할당, Getter/Setter를 1줄로 압축. 데이터 완전성 보장.
*   **Null Safety (`?`):** 컴파일 타임에 NPE 원천 차단.
*   **JPA Entity vs `data class`:** 일반 객체는 `data class`가 표준이나, JPA 엔티티에선 자동 생성된 `equals/toString`이 지연 로딩(N+1)을 터뜨릴 수 있어 일반 `class` 사용 권장.

## 📝 [2일차 추가] 상속과 인터페이스, 그리고 코틀린 문법
*   **상속/구현 통일 (`:`):** 자바의 `extends`, `implements`를 `:` 하나로 통합.
*   **생성자 괄호 `()`:** 부모가 클래스면 생성자 호출(`: Base()`), 인터페이스면 없음(`: Base`). 시각적 구분 명확.
*   **`override` 강제:** 재정의 시 `override` 키워드 강제. 미부착 시 컴파일 에러로 규약 준수 명확화.
*   **단일 표현식 함수 (`=`):** 함수가 단일 결과값으로 귀결될 때 `{}`와 `return`을 생략하고 `=`로 직관적으로 표현.
*   **타입 추론 (Type Inference):** 컴파일러가 반환값을 보고 타입을 완벽히 유추할 수 있을 때, 함수의 반환 타입(`: Type`) 선언을 생략 가능.
*   **우아한 에러 처리 (`runCatching` & `Result`):** 자바의 `try-catch`로 예외를 던지는(Throw) 흐름 제어 대신, 에러를 값(Value)으로 취급하여 `Result` 객체(Success/Failure)에 담아 반환하는 함수형 에러 처리 방식.
*   **외부 API 통신 (RestClient와 동기/비동기):** 
    *   `WebClient` 대신 동기식이면서 체이닝 문법을 제공하는 `RestClient`가 현대적 표준.
    *   가상 스레드(Virtual Threads)를 켜면 블로킹 코드가 논블로킹처럼 동작하여 고성능 발휘.
    *   자바의 `CompletableFuture` 체이닝 대신 코틀린의 `suspend`(코루틴)를 사용하여 비동기 코드를 동기 코드처럼 쉽게 작성.
*   **주 생성자 파라미터와 클래스 프로퍼티:** 주 생성자(`()`) 내부에 `val`/`var`를 붙이면 멤버 변수(프로퍼티)가 되지만, 안 붙이면 단순 매개변수(생성 시점에만 사용)가 된다. 이 매개변수를 이용해 클래스 내부(`{}`)에서 별도의 멤버 변수를 초기화할 수 있다.
*   **컬렉션 리터럴과 중위 함수 (`mapOf`, `to`):** `mapOf("key" to "value")` 처럼 `to`라는 중위 함수(Infix function)를 사용하여 직관적이고 간결하게 Map 생성이 가능하다.
*   **실무형 Null 처리 (엘비스 연산자 `?:`):** 코틀린 실무에서는 `!!`(Not-null 단언) 사용을 극도로 피한다. 대신 `?: throw Exception()` 형태의 엘비스 연산자를 사용하여 Null 발생 시 명시적인 예외나 기본값을 던지는 안전한 패턴을 사용한다.
*   **Reified 타입 파라미터 (스프링 코틀린 확장):** 스프링이 제공하는 코틀린 확장 함수(`import org.springframework.web.client.body`)를 사용하면, `.body(AiResponse::class.java)` 같은 자바식 클래스 전달 대신 `.body<AiResponse>()` 처럼 코틀린의 제네릭(`Reified`) 문법으로 훨씬 우아하게 작성할 수 있다.
*   **후행 람다 (Trailing Lambda):** 함수의 마지막 매개변수가 함수(람다)일 경우, 소괄호 `()` 바깥으로 중괄호 `{}`를 빼서 작성할 수 있다. 만약 람다가 유일한 매개변수라면 `()` 자체를 생략할 수 있어 코드가 매우 깔끔해진다.
*   **Sealed Class (봉인된 클래스):** 자바의 Enum이 '값'을 제한한다면, Sealed Class는 '하위 클래스의 종류'를 엄격하게 제한(봉인)한다. 컴파일러가 모든 자식 클래스를 100% 알고 있으므로 `when` 구문과 결합할 때 `else` 브랜치 없이도 완벽한 분기 처리가 가능하며, 예외(Exception) 계층이나 상태(State)를 표현할 때 실무에서 가장 사랑받는 패턴이다.
*   **예외 전파와 아키텍처 응집도:** 인프라 통신(Client) 계층에서는 원시적인 HTTP 에러를 비즈니스 의미가 담긴 커스텀 예외(Sealed Class)로 '번역(Translate)'하여 던지는 역할까지만 수행한다. 인프라적 재시도(`@Retryable`)는 이 계층에서 캡슐화한다. 반면, 최종 실패 시 롤백을 할지 기본값(Fallback)을 저장할지 결정하는 비즈니스 '흐름 제어(Handling)'는 `Service` 계층이 `runCatching`과 `Result` 객체를 통해 담당해야 계층별 응집도가 극대화된다.

## 🤔 [생각해볼점] 예외(Exception) vs Result 객체 논쟁 요약

Kotlin 실무 및 면접에서 등장하는 **"에러를 예외(Exception)로 던질 것인가, Result(또는 Either) 값으로 반환할 것인가?"**에 대한 아키텍처 트레이드오프.

### 1. 논쟁의 배경
*   Kotlin은 Java의 `Checked Exception` 강제가 람다/스트림과 맞지 않아 이를 폐지(Unchecked로 통일)함.
*   반면 Rust, Scala 등 함수형 패러다임의 "에러도 정상적인 리턴값의 일부"라는 철학이 유입되면서 Kotlin 1.5부터 `Result` 객체가 표준으로 도입됨.

### 2. 전통파 (예외 기반)
*   **Spring 생태계와의 호환성:** `@Transactional` 롤백, `@ControllerAdvice`, AOP, `@Retryable` 등 Spring 생태계 자체가 예외가 '던져진다'는 전제로 설계됨.
*   **보일러플레이트 감소:** 일일이 Result를 풀어서 위로 전달할 필요 없이 자동으로 상위로 전파됨.

### 3. 함수형파 (Result 기반)
*   **타입 가시성(Visibility):** `fun callApi(): Result<T>` 처럼 함수 시그니처만 봐도 실패 가능성을 명확히 알 수 있음 (예외는 내부를 까봐야만 알 수 있음).
*   **흐름 제어:** 외부 API 타임아웃 같은 '예상 가능한 실패'는 비즈니스 로직의 일부이므로 값으로 다뤄야 안전함.

### 4. 실무적 절충안 (베스트 프랙티스 & 면접 모범 답안)
> "Spring의 트랜잭션 관리나 AOP와 자연스럽게 맞물리기 위해 **Spring 프로젝트에서는 예외 기반 방식이 합리적인 기본값**이라고 봅니다. 
> 다만, 외부 API 통신처럼 실패가 '예상 가능한 정상 흐름'인 경계 지점에서는 **`Result`로 명시적 표현**을 하는 것이 가독성과 안정성 면에서 유리합니다.
> **결론:** 어댑터(Client) 계층에서는 예외를 던지고, 이를 호출하는 서비스(Service) 계층에서 `runCatching`으로 잡아 `Result`로 제어하는 절충안을 선호합니다. 단, `@Transactional` 안에서 롤백이 깨지지 않도록 처리 못한 예외는 다시 던지는(`throw`) 것에 주의해야 합니다."
